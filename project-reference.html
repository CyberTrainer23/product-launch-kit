Overview

What is this kit?

The Product Launch Kit is a structured Claude Code project that automates and standardizes BSN's product go-to-market process. Rather than starting from scratch on every launch, the kit guides a product owner through a consistent sequence of phases — each producing a versioned, BSN-branded HTML artifact that can be shared, reviewed, and fed into the next phase.

📋

What it produces

Versioned HTML outputs for every phase: Foundation Documents (Overview, PRD, PDD), a clickable prototype, a Now/Next/Later roadmap, user stories (Jira-ready), marketing messaging, objection handling guides, and demo scripts in three variants.

🔁

How it stays current

Outputs are versioned (Option A — new file on each meaningful update). A staging system prevents accidental writes. Feedback loops connect phases — a prototype update automatically surfaces cascade changes to the roadmap and foundation docs.

🏗️

Where it lives

A Claude Code repo — markdown files that Claude Code reads and orchestrates. Not the BSN skills library. Brand standards are fetched from the BSN MCP server at runtime, with a local fallback if unavailable.

🧩

Who it's for

Any BSN product owner, PM, or marketer running a launch. The kit works for a full launch sequence run start-to-finish, or for individual phases run on their own when only a specific output is needed.

Modes

How it works — two modes

At kickoff, Claude Code asks whether you want to run the full launch sequence or just a specific phase. Both modes are fully supported.

Full launch

Full launch sequence

Runs every phase in order from the questionnaire through GTM. Always pauses at Phase 00 (Foundation Documents) so you can review before continuing.

  1. Document upload (optional)
  2. Questionnaire — Sections A–D
  3. Phase 00 — Foundation Documents → pause
  4. Phase 01 — Prototype
  5. Phase 02 — Roadmap
  6. Phase 03 — Research (opt-in)
  7. Phase 04 — User Stories (skippable)
  8. Phases 05 / 06 / 07 — GTM (parallel)
À la carte

À la carte — pick a phase

Run any single phase independently. Claude Code asks for whatever context it needs at entry — either upload existing files or answer targeted questions. Never refuses to run a phase due to missing upstream work.

  • 00 — Foundation Documents (most common standalone)
  • 01 — Prototype
  • 02 — Roadmap
  • 03 — Research (any sub-phase)
  • 04 — User Stories
  • 05 — Marketing Messaging
  • 06 — Objection Handling
  • 07 — Demo Script
Session resumption: At the end of Phase 00, your questionnaire answers are saved to questionnaire-responses.md. When you return in a new session, Claude Code detects this file and pre-fills context automatically — you won't repeat the questionnaire.
Sequence

Phase flow

Each phase builds on the last. Feedback loops run in both directions — a prototype version bump pushes updates back to Foundation Documents and the Roadmap. GTM findings can loop back to User Stories and the Roadmap.

Questionnaire + document upload

Sections A–D. Upload existing docs for pre-fill. One question at a time — every answer confirmed before moving on.

Produces: confirmed answers for Phase 00

Phase 00 — Foundation Documents

Overview v1, Milestone PRD v1, and PDD v1 — all three in a single tabbed HTML file. Saves questionnaire-responses.md for session resumption.

Produces: overview-v1.html · prd-v1.html · pdd-v1.html · questionnaire-responses.md
⏸ Always pauses here — user confirms before continuing

Phase 01 — Prototype hub

Clickable HTML prototype with all required states (entry, core interaction, confirmation, error). The prototype is a hub — version bumps cascade back to Foundation Docs and the Roadmap. All changes go through the staging system before any file is written.

Produces: prototype-v1.html (and subsequent versions)

Phase 02 — Roadmap

Now / Next / Later roadmap. The Now column is anchored by what's in the prototype. Living document — updated in place, not versioned.

Produces: roadmap.html

Phase 03 — Research opt-in

Five sub-phases: Persona Analysis (03-1), Competitive Analysis (03-2), SWOT+TOWS (03-3), Gap+Premortem (03-4), and Unified Review (03-5). The Unified Review produces Accept / Accept Risk / Reject decisions. Accepted risks carry forward to Phases 05 and 06 via risks-log.md.

Produces: persona-analysis.html · competitive-analysis.html · swot-tows.html · gap-premortem.html · unified-review.html · risks-log.md entries

Phase 04 — User Stories skippable

BDD format (Given/When/Then), Epic → Feature → Story hierarchy, scoped to the Now column. Stories are reviewed in Confluence before Jira import. GTM phases do not depend on this phase.

Produces: user-stories.html
Phases 05, 06, and 07 are independent peers — no dependency on each other. Can run in parallel or standalone.

05 — Messaging

Market track + channel enablement track. Positioning statement, proof points, message pillars, MSP elevator pitch, message matrix.

marketing-messaging.html

06 — Objections

SMB track + MSP track. Frequency-sorted (High/Medium/Low), minimum 8 per track, risk flag integration.

objection-handling.html

07 — Demo Script

Discovery (~15 min), Full Demo (~45 min), Client Enablement (~25 min). Prototype screen references throughout.

demo-script.html

Research — Phase 03 (opt-in)

6 files
phase-03-intake.md research Section E questions (E1–E12). Only asked when Phase 03 runs and the PO opts in. Covers competitive intelligence, win/loss patterns, customer switch reasons, and research depth. Lives here only — never duplicated in the questionnaire.
03-1-persona-analysis.md research Empathetic Researcher persona. Runs each BSN persona against approved upstream documents, produces finding tiles (one per persona — never collapsed), classification layer (Value Statement Miss / Gap / Risk / Suggested Update), severity badges (Critical / Moderate / Minor), cross-persona patterns, and a review summary after the Unified Review completes.
03-2-competitive-analysis.md research Intelligence Analyst persona. Web-researches each competitor across 6 dimensions, assigns confidence levels (Confirmed / Inferred / Unverified) to every claim, cross-checks PO inputs and flags conflicts, produces a competitive positioning SVG map and intelligence summary. Feeds strengths/weaknesses into 03-3 and competitive signals into 03-4.
03-3-swot-tows.md research Clear-Eyed Strategist persona. Produces a sourced SWOT (weaknesses shown at equal weight to strengths — no softening) and four TOWS strategy options (SO/ST/WO/WT), each scored on impact and feasibility. Options, not recommendations.
03-4-gap-premortem.md research Risk-First Thinker persona. Produces gap findings and a structured premortem — failure scenarios written in past tense across 5 categories, scored on likelihood (×0.40) and impact (×0.60), top 5 ranked with early warning signs required for each.
03-5-unified-review.md research Decision tool. Consolidates all Phase 03 findings in a tabbed interactive HTML document. PO-only decisions: Accept / Accept Risk / Reject. Identity tracking and stakeholder comments per finding. Meeting summary panel for live review sessions. Writes to risks-log.md on Accept Risk. Triggers staged-changes.md entries for accepted findings that affect prototype scope. Produces a post-review summary for SharePoint.

Core decisions

Key concepts to understand

A few design decisions that run across the entire skill. Understanding these will make the repo make sense.

Versioning

Option A — new file on each update

Four primary artifacts version this way: Overview, PRD, PDD, and Prototype. Each meaningful update produces a new file (prototype-v2.html, not an edit to prototype-v1.html). Previous versions are always preserved. Minor edits (typos, copy tweaks) do not trigger a version bump.

Staging

Prototype changes go through staging

Claude Code never writes a new prototype version automatically. All changes — from any source — are added to staged-changes.md first. The user reviews the full list and explicitly approves before any file is written. "Actually never mind" removes the entry immediately — nothing is written.

Hub

The prototype is a hub

A prototype version bump is never just a prototype change. It cascades upstream to the Roadmap (scope changes affect Now/Next/Later) and Foundation Documents (Overview, PRD, PDD may need updates). Claude Code surfaces these cascades and asks before acting.

Feedback loops

Phases flow both ways

03-5 Unified Review → Prototype. Prototype → Foundation Docs + Roadmap. GTM phases (05/06/07) → User Stories + Roadmap. Dev track (future) → Prototype + User Stories + Roadmap. No update is automatic — Claude Code always asks before writing.

Brand

MCP fetch → fallback → minimum

Before every HTML output: (1) fetch bsn-tools:bsn-brand via MCP. (2) If unavailable, read bsn-brand-fallback.md. (3) If both unavailable, apply minimum tokens and flag in the version header comment. Outputs using the fallback are clearly marked.

Export

Tabbed HTML · paginated PDF

When two or more documents are exported together, HTML uses a tabbed interface (one tab per document). PDF uses a page break and a document header before each section. Applies to Phase 00 Foundation Documents and any other combined export.

Setup

Getting started

How to set up the kit in Claude Code and run your first session.

1

Clone the repo

All 28 files live in a single directory. Create a logs/ subdirectory with a .gitkeep file to reserve the folder for generated runtime files (risks-log.md, staged-changes.md, questionnaire-responses.md).

2

Open in Claude Code

Open the repo directory in Claude Code. CLAUDE.md is the entry point — Claude Code reads it first on every invocation. No additional configuration is required beyond the repo being open.

3

Verify the MCP connection

The kit fetches brand standards from bsn-tools:bsn-brand at runtime via MCP server https://mcp.dev2.api.pii-protect.com/mcp. Confirm this is accessible from your Claude Code environment. If it isn't, the kit will fall back to bsn-brand-fallback.md automatically — outputs will still be on-brand but should be verified before sharing.

4

Start a session

Tell Claude Code you want to use the Product Launch Kit. It will read CLAUDE.md and ask: "Are you running a full launch sequence, or do you need a specific phase on its own?" — answer that question to get routed into the right flow.

5

First-time: try Foundation Documents standalone

If you're new to the kit, the best first run is: à la carte → Phase 00 — Foundation Documents. Upload an existing product brief if you have one, answer the questionnaire, and see the tabbed Overview/PRD/PDD output. This gives you a feel for the kit before committing to a full launch sequence.

6

Returning to a session

If you ran Phase 00 previously, questionnaire-responses.md should exist in your working directory. When you start a new session, Claude Code detects it and asks if you want to pre-fill context from it. Say yes — it prevents you from repeating the full questionnaire.
