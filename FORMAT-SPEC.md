# FORMAT-SPEC.md — Output Format Specification
## BSN Product Launch Kit · Claude Code deployment

This file is the single reference for all output format decisions across the Product Launch Kit. Phase files and example files point here rather than re-specifying format inline. If a format decision changes, it is logged in `KIT-CHANGELOG.md` and updated here first — the change then applies to all future generations of that document type.

Read this file before generating any output. It supersedes any inline format notes in individual phase files.

---

## Universal rules — apply to every output

These rules apply to every HTML file generated by this kit, without exception.

### File format
- Single-file HTML, self-contained — no external dependencies except Google Fonts CDN
- Downloadable and renderable as PDF without layout breakage
- All CSS inline or in a `<style>` block in `<head>` — no separate stylesheet files

### Version header comment
Required as the first lines inside every generated HTML file:

```html
<!--
  BSN Product Launch Kit
  Document: [document name]
  Version: v[N]
  Generated: [YYYY-MM-DD]
  Reason: [first generation | accepted finding: {title} | prototype v{N} update | PO request]
  Previous version: [filename | none]
  Brand source: [MCP: bsn-tools:bsn-brand | fallback: bsn-brand-fallback.md | minimum tokens]
-->
```

### Brand standards
Fetch `bsn-tools:bsn-brand` via MCP before generating. Fall back to `bsn-brand-fallback.md` if unavailable. See `CLAUDE.md` — Brand standards section.

### Header (every file)
- Background: Navy `#082f49` with hex dot texture
- Logo placeholder (top left) — reference `BSNMain_W.png`
- Document title: Jubilat Bold (fallback: Georgia serif), white, centered or left-aligned
- Subtitle / descriptor: Montserrat, seafoam `#c1ddd8`, smaller
- Pill badge (top right): "Internal use only" — navy fill, electric mint border
- Electric accent line: 2px `#01ffe1` directly below header

### Footer (every file)
- Background: Navy `#082f49`
- Electric accent line: 2px `#01ffe1` directly above footer
- Left text: document name or "Breach Secure Now — Product Launch Kit"
- Right text: "Internal use only — confidential"
- Font: Montserrat 11px, seafoam `#c1ddd8`

### Body background
- Default: `#f0f8fb` (slide background) for text-dense documents
- White `#ffffff` for card-heavy layouts
- Dark/light sandwich encouraged: navy header + navy footer, light body

### Inline callout types (universal)
These visual treatments are consistent across all document types:

| Tag | Background | Border | Text color | Use |
|---|---|---|---|---|
| `[ASSUMPTION]` | `#FAEEDA` (amber 50) | Left: 3px `#EF9F27` | `#4c5454` | Unconfirmed information |
| `[GAP]` | `#FAEEDA` (amber 50) | Left: 3px `#EF9F27` | `#4c5454` | Unaddressed gap flagged for PO |
| `[CONFLICT]` | `#FAECE7` (coral 50) | Left: 3px `#D85A30` | `#4c5454` | Conflicting inputs surfaced |
| `[RISK FLAG]` | `#FAEEDA` (amber 50) | Left: 3px `#BA7517` | `#4c5454` | Risk from risks-log.md |
| `[PRE-FILLED]` | `#E1F5EE` (teal 50) | Left: 3px `#308c83` | `#4c5454` | Answer sourced from uploaded doc |
| `[PENDING UPDATE]` | `#FAEEDA` (amber 50) | Left: 3px `#EF9F27` | `#4c5454` | Deferred cascade update |

All callouts: Montserrat 11px, padding 8px 12px, border-radius 0 6px 6px 0, margin 8px 0.

### Multi-document export
When two or more documents are combined in one file:
- **HTML:** Tabbed interface — one tab per document. Tab bar is sticky at the top of the content area. Tab labels match document names exactly.
- **PDF:** Page break before each document. Document header at top of first page of each section (e.g. "Product Overview", "Milestone PRD"). BSN running header (thin teal line + small BSN mark) on subsequent pages within that document.

## Document-type specifications

### Phase 00 — Foundation Documents (overview-v1.html, prd-v1.html, pdd-v1.html)

**Default output:** Single tabbed HTML file containing all three documents
**Tabs:** Overview · Milestone PRD · PDD (in this order, always)
**Versioning:** Option A — new versioned file on each meaningful update

#### Tab: Overview
**Persona:** Direct + Human
**Sections (in order):**
1. Header — product name, one-sentence value prop, version badge
2. The problem — 2–3 sentences, real-world scenario, specific not generic
3. The solution — earns its place after the problem is established
4. Who it's for — persona cards (one per target persona, pulled from bsn-personas.md)
5. How it works — 3–4 steps, plain language, no jargon
6. What's different — 2–3 differentiators vs. primary alternatives
7. What's next — one sentence pointing to the PRD or prototype for detail

**Layout:** Single-column, generous whitespace. No sidebars. No tables.
**Typography:** Georgia for section headings, Montserrat for body. Never swapped.
**Cards:** Persona cards use BSN card styling — white background, teal glowing border.

#### Tab: Milestone PRD
**Persona:** Precision PM
**Sections (in order):**
1. Header — product name, version, status badge (Draft / In Review / Approved)
2. Problem alignment — problem statement, evidence, user impact
3. Goals and non-goals — two-column layout, equal visual weight
4. Key logic — numbered rules, ASSUMPTION tags on unconfirmed behaviors
5. Key flows — inline SVG flow diagrams + edge case decision table (Trigger / System behavior / Notes)
6. Launch plan — milestones table (milestone / owner / date / status)
7. Open questions — numbered, each with owner and due date

**Layout:** Single-column with two-column sections for Goals/Non-goals and Milestones.
**Special elements:**
- Flow diagrams: inline SVG, linear steps as nodes with arrows, decision points as diamonds, branching paths color-coded (amber for branch, red for destructive, green for success), step numbers as navy circles
- Edge case table: renders directly below each flow diagram it belongs to
- ASSUMPTION tags: inline callout style per universal rules above
- Status badge: pill — Draft = gray, In Review = amber, Approved = teal

#### Tab: PDD
**Persona:** Confident Strategist + Brand Voice Guardian
**Sections (in order):**
1. Header — product name, category, tagline
2. Positioning statement — internal only, full template (For / who / is a / that / unlike / we)
3. Market context — why now, compliance or market driver
4. Target audience — primary and secondary, named specifically
5. Key capabilities — 3–5 capabilities, each with: name, one-sentence description, buyer benefit
6. Competitive positioning — names competitors directly, clear differentiation per competitor
7. Pricing and packaging — tier names, what's included, positioning of each tier
8. Launch messaging — headline, subheadline, 3 message pillars
9. Channel notes — MSP-specific selling guidance

**Layout:** Single-column with capability cards (teal top-strip card style).
**Tone guardrails:** No hedging. Names competitors. Positioning statement is internal — does not appear in external copy.

### Phase 01 — Prototype (prototype-v1.html)

**Versioning:** Option A — new versioned file on each meaningful update, governed by staging pattern
**Single-file HTML:** Clickable, self-contained, simulates navigation between states

**Required states:**
1. Entry point — how the user reaches the feature
2. Core interaction — the main action
3. Confirmation / feedback state
4. Error / validation state

**Layout rules:**
- Role indicator visible on every screen: label showing which system role is being shown (e.g. "Viewing as: Manager")
- State navigation: tab bar or sidebar showing all available states, current state highlighted
- Screen label: small pill badge in top-left of each screen showing screen name (used for prototype references in Phase 04 and 07)
- No final copy — CTAs and field labels are functional but not locked messaging

**Internal badge:** "Prototype — not final" pill badge in header, electric mint border

### Phase 02 — Roadmap (roadmap.html)

**Versioning:** Living document — updated in place, no versioning
**Layout:** Three-column (Now / Next / Later), equal column widths, responsive

**Column headers:**
- Column label: Jubilat Bold, navy, centered
- Horizon note: Montserrat 11px, charcoal, centered below label
- Thin teal separator line below column header row

**Cards:**
- Background: white
- Border: teal glowing border (`box-shadow: 0 0 0 1px #308c83, 0 0 8px rgba(48,140,131,0.15)`)
- Border-radius: 8px
- Padding: 16px
- Stack order within card: Phase tag → Card title → Description → Persona tags → (optional) Dependency note → (optional) Risk flag

**Phase tags:** Pill, solid color per locked table in `02-roadmap.md`, font-size 11px, uppercase, letter-spaced, white text, border-radius 999px. One tag per card maximum.

**Persona tags:** Smaller pills, navy fill, seafoam text, font-size 10px. Multiple allowed per card.

**Risk flag:** Inline amber callout per universal rules. Appears at bottom of card.

**Uncertainty handling:** Placeholder card ("To be scoped") renders in light gray, dashed border, italic title. Uncertainty note renders below the roadmap as a single line in charcoal italic.

### Phase 03 — Research outputs

#### 03-1 Persona Analysis (persona-analysis.html)
**Persona:** Empathetic Researcher
**Layout:** One section per persona. Each persona section contains:
- Persona card header (name, role, quote in first-person voice)
- Signal strength indicator (Confirmed / Inferred / Speculative)
- Finding tiles (one per finding, one per persona — not collapsed across personas)
- Each tile: finding type badge → severity badge → finding text → suggested solution → mitigation → (if applicable) early warning signs

**Finding type badges:** Pill — Value Statement Miss (purple), Gap (teal), Risk (amber), Suggested Update (navy)
**Severity badges:** Pill — Critical (red), Moderate (amber), Minor (gray)

#### 03-2 Competitive Analysis (competitive-analysis.html)
**Persona:** Intelligence Analyst
**Layout:** Competitor-per-section structure. Each competitor section:
- Competitor header: name, category, confidence level badge (Confirmed / Inferred / Unverified)
- Findings table: Claim / Evidence / Source / Confidence
- Cross-check note if PO input conflicts with research finding

#### 03-3 SWOT + TOWS (swot-tows.html)
**Persona:** Clear-Eyed Strategist
**Layout:**
- SWOT: 2×2 grid. Each quadrant: label + sourced bullet list. Weaknesses displayed with same visual weight as Strengths — no softening.
- TOWS: Four strategy options (SO / ST / WO / WT), each scored on Impact and Feasibility (1–5). Score displayed as badge. Options not recommendations.

#### 03-4 Gap + Premortem (gap-premortem.html)
**Persona:** Risk-First Thinker
**Layout:**
- Gaps section: finding tiles per universal format, sorted by severity descending
- Premortem section: failure scenarios in past-tense. Each scenario: category badge → failure description → likelihood score → impact score → weighted total → early warning signs
- Top 5 ranked scenarios highlighted with amber left border

#### 03-5 Unified Review (unified-review.html)
**Layout:** Tabbed — one tab per source document (Persona Analysis / Competitive / SWOT+TOWS / Gap+Premortem)
**Interactive elements:**
- Stakeholder comment field per finding: name + role + comment text + timestamp
- Decision buttons per finding: Accept / Accept Risk / Reject — PO role only
- Confirmation modal on each decision
- Meeting summary panel: toggled from header, shows only findings with comments + decision status
- Scorecard at top: total findings / accepted / risks accepted / rejected

**Post-review summary:** Generated after all decisions made. Separate scrollable section or downloadable summary. Contains: scorecard, document changes queued (green), risks log preview (amber), full decision record with attribution.

### Phase 04 — User Stories (user-stories.html)

**Layout:** Collapsible hierarchy — Epic → Feature → Story
**Collapsible elements:** Epics and Features use `<details>`/`<summary>`. Stories are always visible within an open Feature.

**Story card anatomy (in order):**
1. Story ID badge (pill, navy)
2. Priority badge (Must Have = teal, Should Have = purple, Nice to Have = gray)
3. Role badge (system role, small pill)
4. Story title (Montserrat SemiBold)
5. As a / I want / So that (three lines, each labeled, monospace-style field)
6. Acceptance criteria (Given/When/Then blocks, visually distinct — light teal background)
7. Additional fields (Prototype reference, Dependencies, Risk flags, Notes) — collapsed by default, expand on click
8. ASSUMPTION and GAP callouts per universal rules

**Summary bar:** Fixed or sticky at top of document body: "X epics · Y features · Z stories"

**Epic header:** Jubilat Bold, navy, full-width, teal left border (4px)
**Feature header:** Montserrat SemiBold, charcoal, indented, purple left border (2px)

### Phase 05 — Marketing Messaging (marketing-messaging.html)

**Layout:** Four sections in order, always

**Section 1 — Positioning Foundation**
- Background: navy, white text
- Positioning statement rendered in a card with electric mint left border
- Three proof points as teal checkmark list

**Section 2 — Market Messaging**
- Background: white
- Pill badge: "Market messaging" — teal fill, white text
- Headline: Jubilat Bold, navy, large (28–32px)
- Subheadline: Montserrat Medium, charcoal
- Body copy block: left-border card, Georgia body text
- Three message pillars: teal top-strip cards (one per pillar)
- CTA options: three pill buttons in a row
- ASSUMPTION callouts per universal rules

**Section 3 — Channel Enablement Messaging**
- Background: `#f0f8fb`
- Pill badge: "Channel enablement" — purple fill, white text
- Visually distinct from Section 2 — different background, different badge color
- Elevator pitch: card with purple left border
- Internal talking points: teal checkmark list
- Competitive positioning: collapsible per competitor

**Section 4 — Message Matrix**
- BSN table styling: navy header row (white text), alternating white/`#f0f8fb` rows
- Columns: Audience / Headline / Core value / CTA
- Fixed at bottom of document, always visible (not collapsible)

### Phase 06 — Objection Handling (objection-handling.html)

**Layout:** Two tracks as visually distinct sections, SMB first

**Track header:**
- SMB track: navy background, white Jubilat heading, "SMB" pill badge (teal)
- MSP track: purple background, white Jubilat heading, "MSP" pill badge (electric mint border)

**Objection count summary:** Rendered directly below the main document header, before tracks begin: "SMB: X objections · MSP: Y objections"

**Frequency badge:** Pill at top of each entry — HIGH (teal fill), MEDIUM (navy fill), LOW (gray fill). High-frequency entries rendered first within each track.

**Objection entry:** Collapsible (`<details>`/`<summary>`). Summary line shows: frequency badge + objection text (truncated to ~80 chars). Expanded view shows full anatomy.

**Entry anatomy (expanded, in order):**
1. Objection ID (small gray label)
2. Context note (italic, charcoal)
3. Root concern (italic, slightly indented)
4. Response (body text, Georgia)
5. Proof point (teal left border card)
6. Escalation note (if present — navy left border)
7. Risk flag callout (if present — per universal rules)

### Phase 07 — Demo Script (demo-script.html)

**Layout:** Three variants as navigable sections. Sticky tab bar at top of content area.

**Tab labels:** Discovery · Full Demo · Client Enablement (in this order, always)

**Runtime summary:** Rendered directly below the tab bar before variant content begins: "Discovery: ~15–20 min · Full Demo: ~30–45 min · Client Enablement: ~20–30 min"

**Internal-only sections:** Light gray background (`#f5f5f5`), small "Internal" pill badge (gray fill, charcoal text). Used for: qualification signals (Discovery), prep section (Client Enablement), MSP tips (Client Enablement).

**Demo section anatomy (Full Demo and Client Enablement):**
1. Section title (Montserrat SemiBold, navy)
2. Setup sentence (italic, charcoal — the problem before showing the UI)
3. Demo action (numbered steps, monospace-style)
4. Talk track (Georgia body text, left border card)
5. Pause for reaction indicator (if Yes: small teal "pause" badge)
6. Objection anticipation (if present: amber callout per universal rules)
7. Prototype reference badge (pill, navy fill, links to prototype file)
8. Risk flag (if present: per universal rules)

**Discovery question format:** Numbered list. Each question on its own line, generous line-height. Qualification signals rendered in an internal-only section directly below the questions.

**Prototype reference badges:** Render as inline pills — "prototype-v[N] · [Screen name]" — navy background, seafoam text, font-size 10px.

## Format change protocol

When a format decision changes after FORMAT-SPEC.md is established:

1. Log the change in `KIT-CHANGELOG.md` with the affected document type and what changed
2. Update the relevant section in this file
3. Note the version and date of the change at the top of the affected section
4. All future generations of that document type use the new format
5. Existing generated files are not retroactively updated unless the PO requests it

Format changes do not trigger version bumps on previously generated files. They take effect on the next generation only.
